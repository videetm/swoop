Three architectural approaches

How the app computes and stores your scores under the hood

A

On-Demand Queries

Every time you open the app, it queries HealthKit live and computes scores on the fly. No local database. Simple, always fresh data.

Pros

  • No data layer to maintain
  • Always reflects latest Watch sync
  • Minimal storage use

Cons

  • Slower launch (HealthKit queries)
  • No offline history
  • Battery drain from repeated queries
B

SwiftData + Background Refresh ⭐ Recommended

Computed scores are persisted locally in SwiftData. A background task refreshes once daily (after sleep). HealthKit is source of truth; SwiftData caches results.

Pros

  • Instant app launch
  • Full history & trend charts
  • Works offline
  • Modern Swift/iOS 17+ API

Cons

  • More complex setup
  • Score freshness depends on background task
C

In-Memory + UserDefaults

Queries HealthKit on launch, keeps results in memory via @Observable. Persists only the last 30 days as JSON in UserDefaults. Simple middle ground.

Pros

  • No database schema
  • Easy to reason about
  • Good for prototyping

Cons

  • UserDefaults not built for this
  • Limited history window
  • Hard to migrate later
